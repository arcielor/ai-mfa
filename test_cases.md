# Test Cases - AI/ML-Enhanced MFA System

This document outlines the test cases to verify all core security functionalities of the prototype system in a structured table form.

| Test ID | Test Category | Objective / Description | Test Steps | Expected Outcome |
| :--- | :--- | :--- | :--- | :--- |
| **TC-1.1** | Hashed Password | Successful login with bcrypt hashed credentials | 1. Go to `http://127.0.0.1:5000/`. <br>2. Enter Username: `stud1` <br>3. Enter Password: `stud123` <br>4. Enter Device ID: `device-01` <br>5. Click **Authenticate**. | User is successfully redirected to the OTP Verification page (`/otp`). |
| **TC-1.2** | Hashed Password | Rejection of incorrect password | 1. Go to `http://127.0.0.1:5000/`. <br>2. Enter Username: `stud1` <br>3. Enter Password: `WrongPassword999` <br>4. Enter Device ID: `device-01` <br>5. Click **Authenticate**. | Page reloads displaying the error: `Invalid credentials. Risk: LOW \| Score: 0 \| Attempt 1/5` (or similar). |
| **TC-2.1** | Google reCAPTCHA | reCAPTCHA widget display trigger | 1. Perform 3 consecutive failed login attempts (e.g. inputting incorrect passwords for `stud1`). | On the 3rd failed attempt, the **Google reCAPTCHA checkbox widget** renders right above the "Authenticate" button. |
| **TC-2.2** | Google reCAPTCHA | Block submission without checking checkbox | 1. With reCAPTCHA visible, enter the correct password (`stud123`). <br>2. Click **Authenticate** *without* checking the reCAPTCHA box. | Submission is blocked, showing a warning: `Please complete the reCAPTCHA verification.` |
| **TC-2.3** | Google reCAPTCHA | Successful login with reCAPTCHA checked | 1. With reCAPTCHA visible, enter the correct password (`stud123`). <br>2. Check the reCAPTCHA box and wait for the green checkmark. <br>3. Click **Authenticate**. | User is redirected to `/otp` page. The session counter is reset, and the next login attempt will not show reCAPTCHA. |
| **TC-3.1** | OTP & Countdown | OTP verification success | 1. Log in successfully to reach the `/otp` page. <br>2. Click the **Copy OTP** button to fill the input field. <br>3. Click **Verify OTP**. | Redirection to the Success page displaying: `Login Successful!`. |
| **TC-3.2** | OTP & Countdown | OTP countdown timer displays and ticks down | 1. Log in successfully to reach the `/otp` page. <br>2. Observe the timer widget below the input box. | The clock shows `01:00` and decrements correctly every second (e.g., `00:59`, `00:58`, ...). |
| **TC-3.3** | OTP & Countdown | OTP expiration block and logout redirect | 1. On the `/otp` page, wait for 60 seconds without entering the code. | The timer hits `00:00`, displays `Expired` in red, reloads the page, and redirects to `/` with the message: `OTP expired. Please log in again.` |
| **TC-4.1** | AI/ML Risk Scoring | Unknown device anomaly (Medium Risk) | 1. Go to `http://127.0.0.1:5000/`. <br>2. Enter Username: `stud1` <br>3. Enter Password: `stud123` <br>4. Enter a different/custom Device ID (e.g., `unknown-laptop-xyz`). <br>5. Click **Authenticate**. | The login request encounters a **3-second artificial delay** before loading `/otp`, and the threat status badge on `/otp` shows **🟡 MEDIUM RISK**. |
| **TC-4.2** | AI/ML Risk Scoring | Rate limiting block on brute-force (High Risk) | 1. Submit wrong passwords 5 times consecutively. | The 5th attempt triggers a **🔴 HIGH RISK** block, rendering the `/blocked` template showing a 60-second lock-out timer. |
| **TC-5.1** | Auditing & Logs | Client IP tracking capture | 1. Perform any successful or failed login attempt. <br>2. Navigate to `http://127.0.0.1:5000/logs`. | The logged entry shows the exact IP address (e.g. `127.0.0.1` for local checks), threat indicators, confidence level, and outcome action. |
